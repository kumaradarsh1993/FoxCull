# FoxCull Holistic Audit — July 2026

Scope: full-codebase review (architecture, features, code quality, UX, memory,
security, build/CI) with a **performance focus on the two primary machines**:

| Machine | CPU | GPU | Implication |
|---|---|---|---|
| Alienware 15 R4 | 6C/12T (i7-8750H class) | NVIDIA GTX 1070 | NVENC/NVDEC available (H.264 + HEVC, incl. 10-bit HEVC encode) — currently only used for the H.264 export ladder |
| XPS 13 | 4C U-series | Intel iGPU | Everything CPU-bound hurts 3–5× more here; fork storms and software video decode are the main enemies |

macOS compatibility was audited; per direction, Mac-only issues are filed **P2**.

Companion document: [`BACKLOG.md`](../BACKLOG.md) (prioritized, actionable items).
Fixes already applied during this audit are listed at the end.

---

## 1. Architecture

**Verdict: sound.** The shape of the app is right for its job.

- **Tauri 2 + Svelte 5 + Rust** split is clean: all disk/decode/ffmpeg work is
  behind ~40 typed commands; the frontend never touches paths destructively
  without backend validation (with exceptions — see §5).
- **Per-drive `_FoxCull` library** (catalog.sqlite + thumbs + recycle) keyed by
  **library-relative paths** is the single best architectural decision in the
  codebase: the same SSD culled on the Alienware, the XPS, and a future Mac
  shares one catalog regardless of drive letter/mount point.
- **Browse-in-place, no import** keeps the mental model simple and the failure
  modes small (originals are never modified; exports uniquify).
- **Bounded background pools** (warm pool 1–2 threads, RAW export pool 2,
  single-proxy lock) show hard-won understanding of the real bottleneck:
  disk-seek contention on USB SSDs/HDDs, not CPU. The generation-token
  cancellation pattern (warm_gen / export_gen / frontend epoch counters) is
  applied consistently and correctly.
- **Weak points**: three monolith files carry most of the app —
  `+page.svelte` (~3.0k lines), `EditStudio.svelte` (~4.3k), `commands.rs`
  (~3.5k). Nothing is wrong inside them, but every future feature lands in one
  of these; splitting (stacks logic, culling actions, export dialog, timeline
  engine into modules) is the main maintainability investment worth making.
- **Testing**: Rust has good unit tests for the trickiest parsing (TIFF/IFD
  walk, ICC splice) but they never ran anywhere — the local Windows GNU
  toolchain can't link tests and CI only built release tags. (Fixed: a
  push/PR check workflow now runs svelte-check + cargo check/test.) The
  frontend has zero tests; the stacks grouping logic (`buildRelatedIndex`)
  is pure-function-shaped and would be cheap to cover.

## 2. Performance

### 2.1 What's already good (worth preserving)

- DCT-scaled JPEG decode (1/2..1/8) + EXIF-embedded-thumbnail fast path —
  the right way to thumbnail 50MP phone JPEGs.
- Dev/release cargo profiles that keep image crates at `opt-level=3` — dev
  builds stay usable.
- Viewport-first LIFO thumbnail queue with cancellation; bounded URL memo
  instead of pinned bitmaps; only 6 decoded Focus previews retained. The
  memory discipline notes in `thumbnail-loader.ts` match the code.
- Virtualized grid/strip with rAF-coalesced scroll; snapshot-based undo.
- Warm pool sized for spinning-disk reality (1–2 threads).

### 2.2 Findings (highest impact first)

**PERF-1 · Preview cache is not actually shared between machines, and can be
orphaned by an app update.** Cache keys hash the **absolute path** (plus
mtime/size) with `std::collections::hash_map::DefaultHasher`. The same SSD is
`E:\` on the Alienware, maybe `D:\` on the XPS, `/Volumes/T7` on a Mac — so
every machine regenerates every thumbnail/poster/filmstrip/proxy despite the
documented intent ("generated ONCE and reused on every machine that reads the
SSD"). Separately, `DefaultHasher`'s algorithm is not guaranteed stable across
Rust releases, so a routine toolchain bump can silently invalidate the entire
cache. Fix: key by **library-relative path** with an explicitly chosen stable
hash (e.g. FNV-1a/xxhash). Needs a one-time migration (old entries become
garbage — see PERF-2).

**PERF-2 · The thumbs cache has no garbage collection.** Stale keys (file
edited/moved/renamed, zoom-tier changes, the PERF-1 migration) are "simply
never referenced again" — and never deleted. H.264 proxies are the worst
offender: a full-length 1080p re-encode per HEVC clip, kept forever even after
the clip is culled away (per-file cleanup only runs on in-app delete/move).
`_FoxCull/thumbs` grows without bound on a drive the user is actively trying
to keep tidy. Fix: track last-access (file atime is unreliable — touch mtime on
hit, or a small index) and sweep unreferenced/old entries on folder open or a
periodic pass; show cache size in Settings with a "clear" button.

**PERF-3 · Filmstrip/scrub-strip generation software-decodes the entire clip.**
`fps=` sampling makes ffmpeg decode **every frame** of the source to keep ~40–100
of them. A 5–10 min 4K60 HEVC Osmo clip is minutes of pinned CPU on the XPS 13
(and the scrub-strip generation triggers from a 140 ms hover with Live Scrub
on). Two cheap wins, in order:
1. `-hwaccel auto` before `-i` (D3D11VA/NVDEC on the Alienware, QSV on the XPS,
   VideoToolbox on Mac) — often 5–10× on HEVC;
2. keyframe-only sampling (`-skip_frame nokey` + `fps`) or N seek+grab
   invocations — decodes ~1% of frames; keyframes are visually fine for a
   scrub preview.

**PERF-4 · Edit-mode probe storm (fixed this audit).** Opening Edit mode swept
up to 80 sources through `probe_media_info`, each spawning an ffmpeg process
concurrently on the unbounded blocking pool — a fork storm that visibly stalls
a 4-core machine. Now throttled to 4 in flight. Remaining backlog: persist
probes in the catalog (keyed rel+mtime+size, like `captures`) so Edit mode and
the export dialog stop re-probing every session — probing is also done up to
twice more per clip at export time (`concat_needs_reencode` + `export_reencoded`).

**PERF-5 · Keep-HDR export ignores the GTX 1070.** The HDR passthrough ladder
is `libx265 -preset medium` on CPU — the slowest single operation in the app
(realtime or worse). The 1070 encodes 10-bit HEVC in hardware; try
`hevc_nvenc` with the same color tags first and fall back to libx265. Same
idea for the H.264 proxy transcode (currently libx264-only; the export ladder
already tries NVENC).

**PERF-6 · Mark-keystroke reactivity cost scales with folder size.** A single
rating keypress mutates an item, which invalidates `baseView` (filter + full
re-sort), `relatedIndex` (stacks rebuild — the quadratic bucket fill was fixed
this audit, but the re-rooting pass is still O(n·roots) with per-parent root
lists), then `view` + `sections`. Fine at 1–2k items; on a 5–10k recursive
folder this could produce keystroke lag on the XPS. Before optimizing further,
**measure** — the existing `UI MEM`/`logEvent` plumbing makes it easy to log a
`relatedIndex` build time. Candidate fixes if needed: cache sort keys, skip
re-sort when the mutated field doesn't affect the active sort, and memoize
stack grouping on (paths, names) with only scores recomputed on mark changes.

**PERF-7 · Folder badges can walk enormous trees.** Counts are computed
recursively per expanded node on the warm pool. Expanding `C:\` queues a
recursive media count of *Windows*, *Program Files*, etc. It's off the UI
thread and cached, but it can keep a laptop's disk busy for minutes and the
numbers are meaningless there. Consider skipping known system directories, or
compute badges only for drives the user has opened a folder on.

**PERF-8 · Misc (small, cheap):**
- `LOUPE_MAX`/frontend 1920 is right for both laptops' internal panels but
  soft on a 4K external monitor; derive from the largest connected display.
- The Loupe blur-up placeholder always requests a 320px thumb; if the grid tier
  in use is 192/480 that's an extra cache variant per photo. Pass the grid tier.
- `list_drives` probes `A:`–`Z:` with `is_dir()`; empty card readers can stall
  the call for seconds on some hardware.
- Undo snapshots `JSON.stringify` entire selections for the no-op check —
  select-all(10k) → reject makes two 10k-item JSON strings. Cap or hash.

### 2.3 Memory

**Verdict: healthy.** The 2025 "progressively worse" leak class is genuinely
closed: no decoded-bitmap pinning, bounded memos, generation-token teardown on
folder switch, and a 20 s heap/loader-stats heartbeat into the logfile to catch
regressions. Worst-case pinned decoded memory is the 6 Focus previews
(~60–70 MB) plus whatever the webview's own image cache chooses to keep (which
is its job). Rust-side peak is bounded by pool sizes (2 × ~25 MB RAW reads).
No leak vectors found; `activity` reapers and event unlisteners are cleaned up
correctly. One note: `probes` in EditStudio grows per session without bound —
harmless at realistic folder sizes (~200 B/entry).

## 3. Features & product

The feature set is coherent and unusually complete for its size: cull
(rate/label/flag/tag/filter/sort/group/stacks) → organize (move, trash,
restore) → light edit (trim/subclips/timeline/crop/looks/audio) → export
(stream-copy-first, IG-aware, RAW→JPEG) → cast. The Instagram pipeline notes
(HLG vs PQ tagging, 25 Mbps cap, fps preservation) are correct and better
researched than most dedicated tools.

Gaps that matter for the stated workflow (details in backlog):
- **No filename/text search** in the library — at 10k files, filters alone
  don't cut it.
- **Prepare had no effect on RAW/video folders** (fixed this audit — the
  backend silently filtered them out; RAW culling folders are exactly where
  Prepare matters).
- **Edit preview doesn't play the music lane** — audio clips are export-only,
  which surprises anyone timing cuts to music.
- **Cast chip can go stale** — status is only refreshed on start/stop; if the
  TV drops the session the toolbar still says "casting". `cast_status` exists;
  poll it while the popover claims a connection.
- **Trash has no auto-expiry** — recycle folder grows until manually purged.
- `window.confirm()` gates the JPEG export (see MAC-2 — it can be a silent
  no-op on macOS).

## 4. Code quality

High. Comments explain *why* (often with the war story), naming is consistent,
error handling is deliberate (`Result<_, String>` at the IPC boundary,
best-effort `let _ =` where failure is acceptable). Specific notes:

- `raw.rs` is the standout: documented TIFF walk, bounded against adversarial
  input, tested, with a brute-force fallback.
- Duplication: the `CREATE_NO_WINDOW` Windows block appears ~10×; `uniquify`
  is duplicated across modules; ffmpeg `Command` construction could be one
  helper (`ffmpeg_cmd()`), which is also where `-hwaccel auto` (PERF-3) would
  land once.
- `clean_segments` can emit a zero-length segment when both endpoints are
  negative (filter checks order before clamping) — cosmetic, the trim then
  errors per-segment.
- `eq_folded`'s CSS↔ffmpeg algebra was verified correct (nice work); the
  preview↔export parity design (same control points driving SVG filters and
  ffmpeg curves) is the right pattern.
- Frontend types are honest (no `any` festivals); Svelte 5 runes are used
  idiomatically; effects consistently guard against stale async with epochs.

## 5. Security & robustness

Threat model is a local, single-user tool — the bar is "don't let a webview
compromise or a LAN neighbor touch files the user didn't offer", plus supply
chain. Findings:

- **SEC-1 (P2)** · The v0.2.0 path guardrails (canonicalize → inside active
  root → outside `_FoxCull`) are applied to destructive commands, but newer
  commands bypass them: `cast_start` (registers any path on the LAN media
  server), `export_raw_jpegs`/`raw_embedded_probe` (read/*write* beside
  arbitrary paths), and the read-side commands (`thumbnail`, `loupe_src`,
  `video_*`, `probe_media_info`). Validate all of them against the active
  library root for consistency.
- **SEC-2 (P2)** · Cast media server: binds `0.0.0.0`, tokens never expire,
  allowlist never shrinks, server never stops. Fine on a trusted home LAN,
  sloppy on hotel Wi-Fi (trip laptop!). Cheap hardening: clear the allowlist on
  `cast_stop`, expire tokens after hours, and consider binding to the active
  interface only.
- **SEC-3 (P2)** · CI downloads **unpinned "latest"** ffmpeg builds at release
  time — releases aren't reproducible and the sidecar is a supply-chain trust
  point. Pin exact release tags + verify checksums. (Also note the bundled
  build is GPL: fine to ship as a separate sidecar binary, but the release
  notes/about box should credit FFmpeg and its license — P3.)
- CSP is solid for a Tauri app (no remote script/connect, asset protocol
  scoped). The asset scope does grow to whole drive roots as drives are opened
  (inherent to browse-in-place; accept, but document).
- Catalog durability: WAL + copy-first migration is good. There's no periodic
  backup of `catalog.sqlite`, and it lives on a removable drive that gets
  yanked — a tiny rotating `.backup` on library close would make rating loss
  practically impossible (P2).

## 6. macOS compatibility (all P2 by direction)

- **MAC-1** · Unsigned/un-notarized `.dmg` → Gatekeeper friction (documented
  `xattr` workaround exists). Needs an Apple Developer account + notarization
  step in release.yml. Intel builds intentionally dropped — reasonable.
- **MAC-2** · `window.confirm()` (JPEG export flow) is not reliably implemented
  in WKWebView — on macOS the confirm can resolve falsy without ever showing,
  making "Export as JPEG…" a silent no-op. Replace with an in-app dialog
  (there's already a dialog pattern in EditStudio).
- **MAC-3** · AppleDouble `._*` sidecar files polluting shared exFAT/NTFS
  drives — **fixed this audit** (dotfiles skipped in all walkers).
- **MAC-4** · PERF-1 (absolute-path cache keys) hits hardest on Mac since
  `/Volumes/...` never matches a Windows path — the "cache travels with the
  drive" story only becomes true after that fix.
- Positives: catalog rel-path portability was designed for exactly this;
  `drive_root` understands `/Volumes`; read-only NTFS mounts fall back to
  app-data libraries; HEVC actually plays *better* on macOS (VideoToolbox);
  cast TLS uses Security.framework; Cmd-key shortcuts are handled.

## 7. Fixes applied during this audit

Committed on `claude/fox-cull-audit-kc8iwu`, validated with `cargo check`,
`cargo test` (6/6), `svelte-check` (0 errors, 0 warnings) and a production
frontend build:

1. **Release version stamping** — CI now stamps the tag version into
   `Cargo.toml` as well as `tauri.conf.json`; previously every tagged build's
   window title showed the stale committed version (title reads
   `CARGO_PKG_VERSION` at compile time).
2. **Prepare now actually prepares RAW + video** — new opt-in `heavy` flag on
   `warm_thumbnails`; the explicit Prepare button pre-builds RAW Focus previews
   and video posters (it was silently images-only, i.e. a no-op on RAW/video
   folders). Automatic folder-open warming is unchanged.
3. **Edit-mode probe throttle** — max 4 concurrent ffmpeg probes instead of a
   fork storm of up to 80 (PERF-4).
4. **Stack index de-quadratified** — bucket fill by push instead of per-entry
   array respread (rebuilds on every mark keystroke).
5. **Cross-volume move can't leave duplicates** — if the copy succeeds but the
   source can't be removed, the copy is deleted before reporting failure.
6. **macOS AppleDouble dotfiles skipped** in all media walkers (MAC-3).
7. **Push/PR CI check workflow** — svelte-check + frontend build +
   `cargo check`/`cargo test` on every push (tests previously ran nowhere).
8. **`@types/node` added** — clears the long-standing svelte-check warning;
   checks are now fully clean.
9. Stale `warm_threads` doc comment corrected (claimed 2–4 threads; code
   clamps to 1–2).
