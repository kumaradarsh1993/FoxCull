# RCA: video scrub/preview system feels bad after the nightly.3 rework

**Status: RESOLVED — owner approved the plan on 2026-07-20 and items 1–4 were
implemented (local commit; see `docs/changes/2026-07-20-live-scrub-fix.md`).
Kept for reference until the owner's SSD-vs-HDD live verification passes.**
*Written 2026-07-20 by the base-machine agent. Self-contained: a fresh session
can pick this up without the original conversation.*

## Reported symptoms (owner, 2026-07-20, internal HDD library)

1. Opening a ~2 GB clip in Focus view triggers a **"Building scrub filmstrip"**
   job with an ETA around **1m10s** — *even though Live Scrub is OFF in
   settings*.
2. General video browsing feels much worse than remembered from the pre-rework
   nightlies (which were used off an external SSD).
3. While one of these builds was running against a **17.62 GB HEVC** merge file
   (`*d-merged*`), holding hold-to-delete on it froze the whole app ("not
   responding" — first crash ever observed). *(Fixed separately — see companion
   commit; RCA below explains why the build was complicit.)*

## The regression, precisely

`git show 13ec6a3:src/lib/components/Loupe.svelte` (last pre-rework state) vs
current `main` (`a86ce31`, commit `213902b` "Rework the video preview system"):

- **OLD Loupe** (line ~97/140): `const liveScrub = settings.s.liveScrub; …
  if (liveScrub) { api.videoFilmstrip(...) }` — the dense Focus filmstrip was
  **built only when Live Scrub was ON**. Toggle OFF ⇒ focus view did **zero**
  preview work; you just got a plain seek bar.
- **NEW Loupe** (line ~171): comment literally says *"Scrub filmstrip: ALWAYS
  built for the active clip"* — `api.videoFilmstrip(it.path)` fires
  **unconditionally on every Focus open** of an unbuilt clip. The Live Scrub
  setting is not consulted anywhere in Loupe.svelte anymore.
- Grid-hover strips (Thumb.svelte) still respect the toggle (`if
  (!settings.s.liveScrub) …` unchanged). **Only the Focus path regressed.**

So: the toggle wasn't broken — the rework *removed the gate* on the Focus-view
strip, presumably reasoning "keyframe extraction is cheap now, always-on is
better UX". That reasoning holds on an SSD with H.264; it collapses on the
owner's real setup (below).

## Cost model — why "1m10s" is exactly what the math predicts

Backend (`src-tauri/src/video.rs`, `ensure_sprite`):

- Frame count = `clamp(duration_seconds, 16, 100)` → **any clip ≥ 100 s gets
  100 frames** (a 3-min Dubai clip and a 40-min merge both get 100).
- **Each frame is its own ffmpeg process**: spawn → open the container → parse
  the index (`moov`) → `-ss` seek → decode ONE keyframe → scale → write JPEG.
  Two run in parallel (`SPRITE_PARALLEL = 2`), builds serialized process-wide.
- `extract_frame_at` does **NOT pass `-hwaccel auto`** — the 4K HEVC keyframe
  decode is **pure software**, ~0.3–1.5 s per frame on this class of CPU. (The
  flag exists only in `make_filmstrip_fullscan`, a rare fallback path.)

Per frame on the owner's current setup (internal **HDD**, 2–17 GB 4K60 HEVC):
process spawn (~50 ms) + container open/index parse of a multi-GB MP4 (HDD:
possibly a head-to-tail read if `moov` sits at EOF — phones/editors often put
it there) + one HDD random seek (~10–15 ms but amplified by two processes
fighting for one spindle) + 0.3–1.5 s software HEVC decode + JPEG write to the
cache **on the same HDD** (`_FoxCull/thumbs` lives on the media drive).

→ **0.7–2 s/frame × 100 frames ÷ 2 parallel ≈ 35–100 s. Observed: ~70 s.** The
math and the symptom agree; nothing mysterious is left.

### Why it felt fine before, in one table

| Factor | Pre-rework experience | Now |
|---|---|---|
| Live Scrub OFF | Focus build **never ran** | Runs **always**, first open of every clip |
| Storage | external SSD (parallel random reads ~free) | internal **HDD** (one spindle, seek-bound) |
| Visibility | silent background work | new honest-ETA activity chip **shows** "1m10s" |

All three moved against the user at once. The rework's *extraction method* is
genuinely better (keyframe seeks beat full-decode by 10–100×; the OLD method
with Live Scrub ON would have been *minutes* per clip on this HDD). The
regression is purely **when** it runs, amplified by **where** the data now
lives and by the missing hwaccel.

### Answers to the owner's direct questions

- **"Too many previews?"** Yes-ish: 100 frames/clip is scrub-bar overkill
  (~10 px of timeline per frame on a typical bar). 36–48 is visually
  indistinguishable in use and halves-to-thirds the cost.
- **"Too big previews?"** No. Tiles are 240 px wide; a 100-frame sprite JPEG is
  ~1–3 MB + a tiny JSON sidecar. Disk footprint is not the problem. (Cache GC
  is a real but separate, pre-existing P0 in BACKLOG.md — the cache never
  shrinks, on any version.)
- **"Unnecessary previews?"** **Yes — this is the bug.** With Live Scrub OFF,
  the user asked for none of this work, and pre-rework none happened.
- **Crash connection**: the always-on build had the 17 GB HEVC open in ffmpeg
  when hold-to-delete fired. Windows can't rename an open file → the old code
  fell back to **copying 17 GB** on one HDD inside a **synchronous** Tauri
  command (runs on the main thread) → "not responding". The delete side is
  fixed in the companion commit (no copy fallback, async command, cancel
  background builds first, surface failures). The *build having no business
  running* is this RCA's half of the story.

## Proposed fix plan (awaiting owner confirmation — NOT implemented)

1. **Restore the gate (the actual fix).** Focus view builds the dense filmstrip
   only when Live Scrub is ON. When OFF: show a strip **only if already
   cached** (needs a tiny read-only `video_filmstrip_cached` command mirroring
   the existing `video_scrubstrip_cached`; drafted once, reverted pending this
   approval). Cached-strip display costs nothing and keeps old sessions' work.
2. **`-hwaccel auto` in `extract_frame_at`.** On the Alienware's GTX 1070,
   NVDEC decodes 4K HEVC keyframes ~5–10× faster than software; also aligns
   with BACKLOG P0-2. Turns a 70 s build into ~10–20 s even on the HDD.
3. **Cap dense-strip frames at ~48** (`max_frames` 100 → 48). Visually
   identical scrub preview, ~half the work. (Optionally scale tile count with
   clip length buckets later; not needed now.)
4. *(Design question for the owner)* Even with Live Scrub ON, should the dense
   build wait for **first pointer-down/hover on the seek bar** (intent) instead
   of firing on Focus open? Costs nothing when you only watch; adds a one-off
   delay when you do scrub. My lean: yes, build on intent.
5. **Not proposed**: reverting the rework. The keyframe-seek extractor, the
   cancellation tokens, and the serialized build queue are all strictly better
   than the old full-decode path and should stay.

**Expected outcome after 1–3**: Live Scrub OFF → zero background video work on
open, exactly like the pre-rework builds; Live Scrub ON → strips build in
~5–20 s instead of ~70 s, cancellable, one at a time.

## Verification once approved

- Toggle OFF → open several uncached 2 GB+ HEVC clips → **no** "Building scrub
  filmstrip" activity appears; seek bar still works (plain).
- Toggle ON → same clips → build appears, completes in the predicted window on
  the HDD; scrub preview works; second open is instant (cache hit).
- Owner A/B on the external SSD vs internal HDD to confirm the felt difference.
- Delete a clip mid-build → build cancels, delete succeeds (companion fix).
